# LEDGER-flow

**Lifecycle Evidence, Development Governance & Review — flow**

AI 駆動のソフトウェア開発向けのガバナンスフレームワークです。ルール、ロール定義、テンプレート、CLI ツールを提供し、AI エージェントと人間がプロジェクトライフサイクルを通じてどう協働するかを定義します。

## 課題

AI エージェントにコードを書かせるには、コーディング規約・レビュー手順・ドキュメント基準・エスカレーション方針といったルールが必要です。CLAUDE.md や AGENTS.md、個別のルールファイルに記述しますが、実際にはエージェントは膨大な量の必須指示を読み込んだ上で、その約30%を無視します。

根本的な原因は構造にあります。1つのコンテキストにルールを詰め込みすぎると、どのルールも定着しません。同じエージェントが実装・レビュー・計画・ユーザーへの報告を全て担当すると、コンテキストウィンドウが過負荷になり、遵守率が低下します。

## アプローチ

LEDGER-flow は、ルールを増やすのではなく責務を分割することで対処します。

- **知識の分離** — 実装エージェントに言語ポリシーのルールは不要です。専門の監査ロールが事後にチェックします。各ロールは必要なルールだけを読み込みます。
- **台帳形式の記録** — 全ての判断、レビュー所見、トレードオフの解決をタイムスタンプ付きで記録します。暗黙の上書きはありません。判断の経緯を常に追跡できます。
- **ロードマップ駆動の計画** — 理想体験ドキュメントから逆算して計画し、「ユーザーが意図したものを作れたか」で検証します。
- **エージェントとユーザーのコミュニケーション定義** — 相談テンプレート、エスカレーションルール、自律判断条件により、意思決定の伝え方を構造化します。

## 主要な概念

### 証跡の積層記録

アーキテクチャの選択、レビューの結果、ロードマップの変更といった開発上の判断は、全てタイムスタンプ付きのエントリとして構造化された台帳に記録されます。会計の元帳と同様に、暗黙の上書きはありません。

- 開発ログ — 作業の時系列記録
- レビュー証跡 — 所見、対応計画、フォローアップ結果
- ADR（Architecture Decision Records） — 重要な構造的判断の記録
- ナレッジ — 再利用可能な教訓の蓄積。マスタルールへの昇格パスあり

### ロードマップ駆動の計画サイクル

**理想体験**ドキュメントでプロダクトの目指す姿を定義し、**ロードマップ**でその中のどの部分を次に前進させるかを決めます。

- マイルストーンはユーザーの課題と体験変化に紐付きます
- 計画は理想体験から逆算します。実装の都合からではありません
- 検証は「ユーザーが意図したものを作れたか」で行います

### エージェントとユーザーのコミュニケーション

AI エージェントがプロジェクトの意思決定をユーザーにどう伝えるかを定義します。多くのフレームワークが未定義のまま残している領域です。

- **相談テンプレート** — 軽量版と Full 版の使い分け
- **エスカレーションルール** — 推奨が明確な場合の自律判断条件
- **ロール分離** — 実装、レビュー、計画をそれぞれ別のエージェントが担当

### ロールによる関心の分離

全てのルールを1つのエージェントに読み込ませるのではなく、責務を専門ロールに分割します。

- **Worker** — 最小限のルールで実装に専念
- **コンプライアンス監査** — 言語ポリシー、ドキュメントフォーマット、バックログ検証
- **コード品質監査** — コーディング規約、テスト設計、設計文書との乖離検出
- **Devil's Advocate** — アーキテクチャ判断、構造的整合性（Full review のみ）
- **プランニングリード** — ロードマップ、マイルストーン、ユーザーとの対話

Worker は言語ポリシーのルールを知りません。コンプライアンス監査が事後に違反を指摘します。これは意図的な設計です。全てのルールを背負ったエージェントは、どのルールもうまく守れません。

## 3層の強制メカニズム

- **Tier 1**（pre-commit hooks） — gitleaks + ESLint。機械的に100%強制
- **Tier 2**（CLAUDE.md / AGENTS.md） — プロジェクトルール。LLM の遵守率は約70%
- **Tier 3**（ロールベースレビュー） — 専門コンテキストを持つレビューエージェント群

## ファイル分類

- **マスタルール** — 安定的で更新頻度が低く、常に英語で記述します。ルール、ロール定義、テンプレートが該当します。ledger-flow リポジトリに配置します。
- **プロジェクト進行ファイル** — 開発中に更新され、プロジェクトの Documentation Language で記述します。ログ、レビュー、ロードマップ、設計文書、バックログが該当します。プロジェクトリポジトリに配置します。

## このフレームワークが提供するもの

LEDGER-flow はスターターパッケージです。AI 駆動開発のための進め方、プロセス、ドキュメントテンプレートを一式提供します。`ledger init` を実行すると全てがプロジェクトに転写され、以降はプロジェクトの所有物になります。

- `development-docs/rules/` にはフレームワーク提供のデフォルト設定が含まれます。編集は可能ですが、編集後はフレームワーク新版への機械的な追従ができなくなります。
- プロジェクトの事情に合わせて変更してください。その際、フレームワークが前提とする思想（証跡主義、ロールによる責務分割、テンプレートを通じた構造化された出力）は維持することを推奨します。
- フレームワーク改定への追従方針はプロジェクトのライフサイクル管理の一部であり、このフレームワークの責務外です。

## はじめかた

> LEDGER-flow は初期開発段階です。

```bash
# プロジェクトを作成
mkdir my-project && cd my-project && git init

# ledger-flow を任意の場所に clone（一時的な作業用）
git clone https://github.com/myurait/ledger-flow.git /tmp/ledger-flow

# プロジェクトルートでセットアップを実行
python3 /tmp/ledger-flow/scripts/ledger.py init \
  --doc-lang Japanese --dev-lang TypeScript

# init 完了後、フレームワークのコピーは不要
rm -rf /tmp/ledger-flow
```

`ledger init` はフレームワーク全体をプロジェクトに転写します。init 完了後、プロジェクトは自己完結します。submodule やフレームワークリポジトリへの依存は残りません。

以下の構造が生成されます:

```
my-project/
├── CLAUDE.md                         # リダイレクタ（development-docs/entry-point.md をインポート）
├── AGENTS.md                         # Codex / Copilot 向けハイブリッドエントリポイント
├── .pre-commit-config.yaml
├── .claude/agents/                   # Claude Code サブエージェントラッパー
└── development-docs/
    ├── entry-point.md                # ロールルーター実体（AI の実際のエントリポイント）
    ├── project-policies.md           # プロジェクト固有ポリシー
    ├── .ledger.json                  # フレームワーク派生バージョンのスナップショット
    ├── rules/                        # フレームワーク提供のデフォルト
    │   ├── AI_RUNTIME_RULES.md
    │   ├── language-policy.md
    │   ├── coding-conventions.md
    │   ├── testing.md
    │   ├── development-process.md
    │   ├── roles/                    # 5 つのロール定義
    │   ├── templates/                # レビュー / ロードマップ / 進捗報告テンプレート
    │   └── scripts/                  # CLI ツール（ledger log/review/roadmap/backlog）
    └── project/                      # プロジェクト所有のコンテンツ
        ├── design/
        ├── features/
        ├── requirements/
        ├── logs/
        ├── reviews/
        ├── roadmap/
        ├── decisions.md
        └── knowledge.md
```

プロジェクトルートの外側には実行時依存はありません。`development-docs/rules/` はフレームワーク提供ですが、init 後は完全にプロジェクトの所有物です。必要に応じて変更できます（フレームワーク改定への機械的追従ができなくなる代わりに）。

### CLI コマンド

`ledger` CLI は init 時に `development-docs/rules/scripts/` に転写されます。プロジェクトルートから実行します:

```bash
# 開発ログエントリを追加
python3 development-docs/rules/scripts/ledger.py \
  --base-dir development-docs log add \
  --task "認証フローを実装" --author "your-name"

# レビュー証跡ファイルを作成
python3 development-docs/rules/scripts/ledger.py \
  --base-dir development-docs review create \
  --scope "auth-flow"

# 新しいロードマップファイルを作成
python3 development-docs/rules/scripts/ledger.py \
  --base-dir development-docs roadmap create \
  --scope "next-milestone"

# バックログのフォーマットを検証
python3 development-docs/rules/scripts/ledger.py \
  --base-dir development-docs backlog validate
```

## 開発フロー

全変更に対する6ステップと、大きな作業に対するトリガー制ステップがあります。

**常に実行:**
1. 変更前に既存コードを読む
2. 実装する
3. テストを追加・更新する
4. 開発ログを記録する
5. レビューを実行する（Light: コンプライアンス + コード品質。Full: + Devil's Advocate + プランニングリード）
6. コミットする

**トリガー時に追加:**
- 新機能 / ロードマップ作業 → 理想体験との整合性を確認
- アーキテクチャ変更 → ADR を記録
- Feature / マイルストーン完了 → プランニングリードが進捗を評価

## ツール互換性

LEDGER-flow は特定の AI コーディングツールに依存しません。

- **Claude Code** — CLAUDE.md + .claude/agents/（初期実装の主要ツール）
- **OpenAI Codex** — AGENTS.md 経由でサポート
- **GitHub Copilot** — AGENTS.md + CLAUDE.md の両方を読みます
- **Cursor** — .cursor/rules/ ラッパー（計画中）
- **Windsurf** — .windsurf/rules/ ラッパー（計画中）

## 背景

LEDGER-flow は実際のプロジェクトでの失敗から生まれました。

複数の AI エージェントを使用するプロジェクトで、あるエージェント（Codex）にロードマップの策定を任せました。自律的に作業した後、ターミナルいっぱいの構造化されていないテキストを返し、「方針を決めてください」と求めてきました。内部用語やマイルストーン ID が詰め込まれた、ユーザーにとって判断材料にならないメッセージでした。リポジトリ全体の監査では91件の指摘が出て、そのエージェントはプロジェクトから外されました。

代わりに投入した Claude Code も、同じ言語ポリシーに違反し、同じレビューステップを省略するという同じ失敗をしました。

2つの異なるエージェントが同じ失敗をしたことで、問題がエージェントの能力ではなくガバナンスの構造にあることが明確になりました。コンテキストウィンドウを圧迫する膨大な必須読み込み、規模を問わない全ステップの強制、正本のないルールの重複、実装用ルールとレビュー用ルールの未分離、エージェントからユーザーへの伝え方の未定義。

LEDGER-flow は、その構造的な問題を再設計した成果です。
